Then AI arrived

AI does best on
simple projects

Not necessarily small. Simple. Understandable architecture, predictable APIs, explicit behavior, limited dependencies.

On those, AI can trace execution, suggest fixes, generate tests, refactor confidently. Open a project with hundreds of deps and framework magic, and the AI struggles.

And honestly? So did I.

AI has a context budget

Humans have attention.
AI has context.

Every dependency, every abstraction layer, every hidden implementation spends part of that budget.

Code isn't just written for humans anymore. It's written for machines that help humans.

And that changes the design constraints.

The Gravity Waves story · 2024

A little
spaceship game

You piloted a ship through space — with a twist: you could move backward and forward through time. The game state itself could rewind.

I needed exactly two things:

A template engine

A state store with history

The existing libraries were mature, battle-tested, feature-rich — and huge. They weren't bad. They were solving bigger problems than I had. I needed less.

The dangerous idea

"What if I remove
all the dependencies?"

Not because dependencies are bad. Because this project didn't need them.

The experiments

Could I build a tiny template engine? A tiny state store? Keep everything understandable? Own every line?

That became
Spektrum.

The irony

I just became
a dependency

The moment you publish a library, you create another package. You become another link in someone's trust chain.

So the goal was never "eliminate dependencies." That'd be absurd. The goal became:

"Understand every dependency." — starting with my own.

A shift in thinking

AI helped build it →
I built it for AI

Most frameworks optimize for developers. I wanted to optimize for developers, AI systems, auditing, and testing — all at once.

  • Keep the codebase intentionally small
  • Reduce magic and hidden behavior
  • Make state transitions explicit and behavior observable

If AI can understand the whole framework, it can reason about everything built on top of it.

Back to basics

Do you still
need TypeScript?

If you're coding with AI anyway — for a small library, hand-written .d.ts files (plus a little JSDoc) give you full autocomplete and typed consumers without a compile step.

The usual

.ts → tsc → bundler → dist/

Spektrum

plain .js + .d.ts → ship it

Strip the build step and you can drop the whole thing on a static host and it just works. No pipeline, no dist/, nothing to compromise between source and shipped.

⚠️ honest scope: .d.ts types the boundary, not the implementation — pair with // @ts-check if you want in-body checking. And no — this isn't right for everything. But for prototypes? Wonderful.

Show, don't tell

Live demo

npm i spektrum # 0 runtime deps, no build step

Bringing a time machine

The game mechanic
became a tool

State history. Time travel. Rewind. What started as a gameplay gimmick turned out to be an engineering primitive.

Suddenly Spektrum wasn't just a template engine. It was an experiment in AI-first software design.

Time travel as a testing model

Rewind instead
of rebuild

Humans test like this

run → reset → run → reset → run

What if state is part of the model?

scenario A → rewind → B → rewind → C

No rebuilding. No reinitializing. No starting from scratch every time.

So should we stop using dependencies?

Absolutely not.

The npm ecosystem is one of our industry's greatest achievements. The message is simpler — before adding a dependency, ask:

  • Does Node.js already provide this?
  • Do I really need this functionality?
  • Is the package actively maintained?
  • What does its dependency tree look like?
  • Can I — my team — and AI — audit it?
  • What's the true cost over the lifetime of the project?
Conclusion

Understanding is
valuable again

Software is no longer consumed only by humans. AI reads it, tests it, audits it, extends it. So simplicity matters again — not because complexity is evil, but because understanding is valuable.

Maybe the answer isn't to eliminate dependencies. Maybe it's to understand them better.
