Back in time · 2023

Gravity Waves

I dusted off an old idea: a little spaceship game where you ride gravity waves to jump back and forth through time — the game state itself could rewind. I needed two things: a small templating engine and an event store.

Every library had so much overhead and config, so I did the stupid tinkering thing and tried to build my own.

2023

GPT-4 had just landed — a real step up. But hallucinations everywhere, no reasoning, and tiny context that degraded fast. Just you and a chat window.

2026

Models like Claude Opus 4.8: far more capable — but token-hungry, and you still have to guide them. And npm supply-chain attacks, seemingly every week.

Gravity Waves got stuck in time. But it looked like things had to change.

First, credit where it's due

npm is a success story

Thousands of ready solutions, so you don't reinvent the wheel every time. You stopped coding everything yourself — and trusted the packages were relatively safe.

Routing libraries

Build tools

Database drivers

Testing frameworks

Auth libraries

All the things that normally give you headaches.

Dependencies are not the enemy.

…but lately it looked grim

The supply chain

  • Keys stolen from big and small repos; packages infected
  • Keeping everything up to date — a frustration of its own (Mend, Checkmarx…)
  • Dependencies on long-abandoned projects I never fully controlled
  • Typosquatting — tricking even the AI into installing malicious packages
  • All that package context burned tokens. Wasteful.

So I asked: why do I even have all these dependencies?

Big production projects need rigid, mature libraries for every edge case. My prototypes don't. So — who's pulling in ExpressJS just to serve a static page?

Both render the exact same page. That 3.7 MB is a lot of attack surface for a static page. Maybe these three lines are enough.

A mindset shift

Going AI-first

If I have no dependencies, the AI doesn't have to load all that context. Cheaper. More focused on what actually matters in the project. So how do I make the whole AI workflow faster and cheaper?

Zero deps

nothing to audit, update, or burn context on

Zero build

no bundler, no config files, no GitHub Actions

Pure vanilla JS

kept only .d.ts files for a little TypeScript support

Bonus: merge to main and it's instantly live on GitHub Pages — completely free.

The guinea pig

I wrote a context file

A new model dropped, and I found my broken Gravity Waves. The perfect test: a templating engine with a built-in time-machine — zero deps, zero build. I wrote down what it was, its functionality and the new restrictions, then let the AI work it out phase by phase, to-do by to-do.

# AGENTS.md — driving Spektrum as an AI agent
## Why Spektrum for agents

~1100 LOC, single file — the entire engine fits in
  your context window. One Read and you've grokked it.
setValue('user.email','x') — path-based mutation is
  structured data, easy for an LLM to synthesize.
time-travel is the primitive — try an edit,
  evaluate, roll back. Every move is replayable. ▋

No endless reasoning, no drifting through dependencies, lowest token usage ever. It worked so well it became a real npm package: Spektrum.

Spektrum didn't load.

This deck navigates through Spektrum's own event store — there is no fallback by design. It loads Spektrum from unpkg, so it needs a working network. To fix:

  1. Check you're online and that https://unpkg.com/spektrum@1.1.0 is reachable (some venue wifi blocks CDNs).
  2. If unpkg is blocked, drop a copy of spektrum.js next to this file and change the importmap to "spektrum": "./spektrum.js".
  3. Serve over http://, not file:// — ES modules won't load from file://. python3 -m http.server
  4. Check the console below for the exact error.
